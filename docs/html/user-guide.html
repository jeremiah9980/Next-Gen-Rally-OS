01

First-time setup

One command configures everything — environment file, database, AI key, and demo data. Run it from the repository root. 

# requires Node 20+ and pnpm
pnpm install
pnpm run setup
1
Prerequisites check

The installer verifies Node 20+ and pnpm, and stops with a fix-it hint if either is missing.

2
Database connection

Paste your PostgreSQL connection string (local Postgres, Neon, or Railway). The installer saves it to .env and pushes the schema.

3
AI key (optional)

Paste an Anthropic API key to enable practice-plan generation and player-trend suggestions. The installer validates it live against the API before saving. Skip it and everything else still works.

4
Seed demo data

Say yes to seeding and you get a demo organization, team, season, and a coach login to explore with.

💡

CI or scripted installs: pnpm run setup -- --yes accepts every default and keeps existing values — no prompts.

04

Importing your roster

Open NCS Roster Dashboard in the sidebar. There are two ways in — both land in the same preview, and nothing is written until you click Import.

Fastest Path A

Search NCS Teams

Pull your roster live from playncs.com — no copying.

  1. Enter team name, city, and/or state → Search
  2. Find your team in the results table
  3. Click Pick This Team
  4. The live roster drops into the preview
Path B

Paste Text

For anything the live search doesn't cover.

  1. Copy roster table rows from any NCS page
  2. Paste into the text area → Preview Roster
  3. The parser maps columns by header — or by pattern (jersey #, position, grad year) when there's no header

05

Reviewing NCS changes

The background worker polls your team's NCS page on a schedule and diffs it against your last import. Every difference — a new player, a jersey change, a removal — lands in the Change Review Queue. You decide what's real.

change_detected → pending_review → accepted or ignored
1
Open NCS Change Review

Each card shows the change type (add / remove / update), the player, and the before → after values.

2
Flag for review (optional)

Not sure yet? Move it to pending_review and come back after talking to the family or the association.

3
Accept or ignore

Accept applies the change to your canonical roster and advances the baseline. Ignore dismisses it. Both are final — the queue stays clean.

🔁

Manual re-diff: you don't have to wait for the worker. Paste a fresh roster into the Re-Diff panel on the Roster Dashboard any time to generate change reviews on demand.

07

AI practice planning

Tell the planner when, where, and what to focus on — it drafts a complete, age-appropriate session in two versions.

1
Fill in the practice form

Date, duration, location, and the team focus (e.g. “first-and-third defense, two-strike hitting”). Recent game stats and player notes are pulled in automatically as context.

2
Generate

Claude drafts a full schedule in timed blocks, with drill instructions, player groupings, coach assignments, equipment, and a weather contingency plan.

3
Two versions, automatically

The coach version includes private player notes and development focuses. The player version strips all of that — safe to share with the team and families.

🔐

Privacy by design: private coach notes can never leak — the player version is a separate document generated without them, and the public site only ever sees the player version.

08

Publishing your team site

The Org Builder turns a short YAML config into a provisioned team and a publish-ready site. Start it with pnpm --filter @rally/org-builder dev.

1
Edit the SiteConfig

Organization name and slug, team name, season, age group, and which of the 13 site modules to enable (roster, schedule, tournaments, stats, fundraising…).

2
Validate & preview

Instant validation with a preview of exactly which modules will render and the publish target/domain.

3
Provision

Creates the Organization → Team → TeamSeason records (idempotent — safe to run again after edits).

4
Org Kit

A starter brand kit generated from your config: a color palette derived from your org, an SVG logo with your team initials, four ready-to-paste social posts, and a Vercel deploy checklist with your slug and domain pre-filled. Everything is click-to-copy.

5
Deploy the public site

Follow the checklist: one Vercel project per team, root directory apps/public-site, set PUBLIC_SITE_ORG_SLUG, point your domain. Re-deploy whenever you accept NCS changes or update team info.
